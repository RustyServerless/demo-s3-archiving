# Rust contender — `figment-engine-chain`

The **speed-optimal** sibling of [`figment-engine`](../figment-engine/README.md).
Same benchmark contract — archive ~3 000 objects (~14.65 GB) under a prefix
into one flat **STORED ZIP64**, in a single arm64 / `provided.al2023` Lambda —
but a different goal. Where `figment-engine` optimises `run_price_usd`
(`memory_mb × duration`) and stays throttle-safe on one key, this design
chases **wall-clock**: it moves **almost zero image bytes through the Lambda
ENI at all**.

It depends on the `figment-engine` crate by path and reuses its ZIP64 encoder,
CRC decoding, and S3 plumbing; it adds only its own planner and executor.

## The core idea: copy *everything*

`figment-engine` already keeps the **bigs** off the ENI (server-side
`UploadPartCopy`) and streams only the **smalls** — ~7.5 GB still crosses the
Lambda. The question this contender asks: **can the smalls be copied too?**

If every body — big and small — reaches the archive by server-side copy, the
Lambda carries no bodies at all, and the run is no longer bounded by ENI
bandwidth (~100 s+ for any streaming design). It's bounded instead by how fast
S3 will accept control-plane calls. That's a different, much lower floor.

There's exactly one body that *can't* be copied: the very first. A valid
STORED ZIP has **no preamble** — byte 0 must be the first local header's
`PK\x03\x04` signature, not a raw body. A pure copy would put a JPEG at offset
0 and destroy the signature, and a part can't mix Lambda-written header bytes
with copied body bytes. So entry 0 concedes **one 5 MiB bootstrap read** (its
header plus the first 5 MiB of its body, uploaded as a real part), and its
remainder — and every other body in the archive — is copied. The honest
headline is therefore "**one read**", not "zero reads": a single 5 MiB read
against a 15.7 GB archive.

## The challenges

Copying the smalls runs straight into the same 5 MiB floor that shaped
`figment-engine`, but harder — because now there's no streaming to lean on.

1. **The floor, with no stream parts.** A small (< 5 MiB) can't be a standalone
   non-last copy part. In the single-MPU design, smalls were *batched into
   stream parts*; here we've sworn off streaming, so that escape is gone.
2. **Headers can't share a part with copied bodies.** A header is
   Lambda-written bytes; a body is a server-side copy; **a single MPU part is
   either generated bytes or a copy, never both**. So a header and its body
   can't be one part — yet in the archive they must be adjacent.
3. **One MPU can't hold it.** A pure-copy archive of more than one entry can't
   be a single MPU at all (proved below) — it forces a multi-MPU structure.
4. **Throttling.** A multi-object, many-MPU design issues *thousands* of
   control-plane calls. The streaming designs avoid this; a copy-only design
   has to issue the calls without tripping S3 `SlowDown`.

## The solution: a chain of per-segment MPUs

### Segments and the trailing-header trick — solves 1 & 2

The unit of work is a **segment**: one floor-anchoring big, followed by its
smalls.

```
  [LFH_big][big]  [LFH_s1][s1]  [LFH_s2][s2] …
```

The big sits at the front and clears the floor by itself as a copy. Each small
is appended as the **floor-exempt last part** of its own step, so it never has
to clear the floor alone. And each body's **header rides as the tail of the
previous step's generated bytes** — the Lambda emits a thin stream of header
bytes and never a body. A big's own header rides the *previous segment's*
tail; only entry 0's header has no predecessor, which is what the bootstrap
read handles.

### Why it must be a chain, not one MPU — solves 3

A segment of one big + *k* smalls is built as a short serial chain of *k*+1
MPUs ("links"). Each link copies the accumulation forward as its first part
and appends the next piece as the floor-exempt last part:

```
  Link 0:  create [LFH_big][big]
  Link 1:  copy-forward  +  append [LFH_s1][s1]   (exempt last part)
  Link i:  copy-forward  +  append [LFH_si][si]   (exempt last part)
```

The links within a segment are serial — each needs the previous link's
*completed* object as its copy source. Different segments are fully
independent and run concurrently. The finished segment objects are then
copy-stitched into the final archive by one flat MPU (segment *k* → stitch
part *k+1*), with the central directory as the exempt last part.

**Why not one MPU per segment?** Because the format forbids it:

> **A pure-copy archive of more than one entry cannot be a single MPU.** Every
> non-last part must be ≥ 5 MiB and is *either* generated *or* copied. Headers
> (~50 B, generated) and bodies (copied) alternate at every entry boundary,
> forcing a part boundary at each header. A sub-floor run is legal only as an
> MPU's *one* last part — so a single MPU places at most one header. The
> multi-MPU chain is **forced by the format**, not chosen.

The closed form: **minimum MPUs = 2n − S** (n objects, S segments). More
segments means fewer total links, so calls are minimised by **maximising S** —
one segment per big, smalls spread one-per-segment ("spread thin"). With the
benchmark's ~1:1 ratio, nearly every chain is depth 2–3 — the shallowest the
floor permits. The structure is provably call-minimal; the remaining gains are
all *operational*.

### Why it doesn't throttle — solves 4

This is the counter-intuitive part, given the `figment-engine` README's caution
that multi-object fan-out "reliably trips SlowDown". The distinction is **rate,
not count**:

- The design issues ~22 k control-plane calls — far more than the streaming
  designs.
- But the links within each segment are **serial** (each waits on the
  previous's completed object), so the achievable *rate* is bounded by chain
  latency, not by how many futures you spawn.
- In isolation the chain sustains only ~1 500–1 600 calls/s — well under half
  S3's ~3 500/s per-bucket `SlowDown` knee. **It cannot issue fast enough to
  trip the throttle**, even at high concurrency.

So a copy-only design is, paradoxically, *gentle* on the bucket — the opposite
of the streaming contenders that 503 under contention. The structural
serialism that bounds its speed is the same thing that bounds its call rate
safely under the knee.

## What actually bounds the wall-clock

Because no bytes cross the ENI (bar the one bootstrap read), the run is bound by
**serial chain latency × per-call round-trip time**, not bandwidth and not the
throttle. The phase breakdown on the benchmark data:

```
PHASE crc_heads        ~1.9 s    HEAD ~3000 objects for stored CRC32 (read-side)
PHASE cd_eager         ~0.04 s   build + upload the central directory up front
PHASE build_and_stitch ~12.2 s   the segment chains + overlapped stitch — the floor
PHASE terminal_complete ~0.5 s   the single CompleteMultipartUpload on 15.7 GB
```

`build_and_stitch` is ~all of it, and it's serial-latency-bound: ~22 k calls
through chains that are ~2–3 links deep, each link a serial
`create → copy → append → complete`. Confirmed by experiment — raising segment
concurrency from 256 to 512 did **not** help (it's not concurrency-starved),
and the AWS SDK's HTTP layer exposes no connection-pool knob that would
(connections are already effectively unbounded). The floor is the round-trip
latency of the serial chains, which is structural.

### Memory is a vCPU dial, not a RAM need

The design does no real compute — no compression (STORE), no body buffering
(copy-only), no hashing (CRC32 comes from S3 HEAD metadata). Peak RAM is
~60–75 MB at *every* memory size. So Lambda memory matters only as the vCPU
lever that drives the 256-wide async reactor. A solo sweep (each tier deployed,
invoked warm, archive re-validated) shows throughput scaling near-linearly with
memory until it hits the ~10.5 s serial-chain build floor:

| Memory | `build_and_stitch` | Total | calls/s | Note |
|---|---|---|---|---|
| 256 MB | ~64 s | ~65 s | ~351 | vCPU-starved — reactor can't service the concurrency |
| 512 MB | ~30 s | ~31 s | ~747 | |
| 768 MB | ~16 s | ~20 s | ~1173 | `memory × duration` (run_price) minimum |
| 1024 MB | ~16.5 s | ~17 s | ~1365 | |
| 1536 MB | ~11.1 s | ~12 s | ~2030 | ~0.5 s above the floor |
| **1792 MB** | **~10.5 s** | **~11 s** | **~2150** | **lowest memory at the latency floor** |
| 2048 MB | ~10.5 s | ~11 s | ~2150 | no gain over 1792 |
| 4096 MB | ~10.4 s | ~11 s | ~2150 | no gain — pure waste |

Two operating points fall out. **1792 MB** is the *speed* choice used here: the
lowest memory that reaches the ~10.5 s build floor, so nothing is paid for vCPU
that no longer cuts duration. **768 MB** is the *price* choice — it minimises
`memory × duration` while still finishing in ~20 s (well under the field), so it
would win a cost-ranked board. This entry optimises for speed.

## How it got here (the optimisation sweep)

Each step was deployed, solo-invoked, and the resulting archive re-validated by
the control Lambda before the next:

| Change | `build_and_stitch` | Total | Note |
|---|---|---|---|
| Simple+correct baseline (end-of-run stitch, 64-wide) | ~27.7 s | ~34 s | First valid copy-only archive |
| Overlapped completion-driven stitch + eager CD | ~32.5 s* | ~36 s | *stitch folded in; not yet the lever |
| Concurrency 64 → 256 | ~13.9 s | ~17.6 s | The big lever — was under-concurrent |
| Per-link deletes off the critical path | ~12.3 s | ~15 s | One round-trip removed per link |
| Cleanup → S3 lifecycle (off the measured path) | ~12.2 s | **~TODO** | Benchmark times the invoke; GC moved out |

Build-correct-first was deliberate: a validated baseline meant every
optimisation could be checked against "still produces a byte-valid archive",
not guessed at.

## Architecture

Same pure-engine / thin-executor split as `figment-engine`:

```
   list files          ┌──────────────────────────────────────────┐
   (name, size) ──────►│  plan_chain  (pure, total, TDD'd)        │
                       │  • partition bigs / smalls               │
                       │  • one segment per big; spread smalls    │
                       │  • emit ordered links per segment        │
                       │  • compute exact ZIP offsets + CD        │
                       └───────────────────┬──────────────────────┘
                                           │  ChainPlan
                                           ▼
                       ┌──────────────────────────────────────────┐
                       │  assemble_chain  (executor, no logic)    │
                       │  HEAD all → CRC32                        │
                       │  build CD eagerly (off critical path)    │
                       │  build segments concurrently (256-wide)  │
                       │    each fires its stitch copy on finish   │
                       │  one terminal CompleteMultipartUpload    │
                       └──────────────────────────────────────────┘
```

Intermediate segment/link objects live under a dedicated **`seg-temps/`**
prefix and are reaped by an S3 **lifecycle rule**, not by the Lambda — so
cleanup never sits inside the benchmark-measured invoke duration.

## Design decisions (summary)

| Concern | Decision | Why |
|---|---|---|
| Move strategy | Copy **every** body server-side | Escapes the ENI bandwidth floor entirely |
| Entry 0 | One 5 MiB bootstrap read | A STORED ZIP has no preamble; byte 0 must be `PK\x03\x04` |
| Smalls below floor | Appended as each link's exempt last part | A sub-floor run is legal only as an MPU's last part |
| Headers | Ride the tail of the previous link's bytes | A part is generated *or* copied, never both |
| Per-segment structure | Chain of *k*+1 MPUs ("links") | Pure-copy >1 entry can't be one MPU (2n−S minimum) |
| Allocation | One segment per big, spread smalls thin | Maximising S minimises calls *and* chain depth |
| Stitch | One flat MPU, completion-driven | Each segment's copy fires as it finishes; one Complete at the end |
| Central directory | Built eagerly after CRCs, uploaded as exempt last part | No body IO to contend with, so nothing needs deferring |
| Throttle-safety | None needed in isolation | Serial chains self-limit to ~1.6 k calls/s, under the knee |
| Cleanup | S3 lifecycle rule on `seg-temps/` | Keeps GC out of the benchmark-measured invoke |
| Memory | 1792 MB / arm64 | No bodies/compute; pure vCPU dial. Lowest tier at the ~10.5 s build floor (see sweep); ~75 MB peak |

## Verification

- **Engine tests** (`cargo test -p figment-engine-chain --features zip_validate`):
  simulate the executor in memory — build each segment object link-by-link,
  stitch, and round-trip through a standard ZIP reader, asserting every part
  clears the 5 MiB floor and extracted content hashes match entry names.
- **Control-Lambda check**: re-reads all 15.7 GB, hashes every entry, asserts
  entry-name == SHA-256(content). Run after **every** optimisation in the sweep
  above — the design has been re-validated end-to-end at each step.
- **No throttling**: zero `SlowDown` across the whole solo sweep, confirming the
  serial-chain rate stays under S3's knee.
